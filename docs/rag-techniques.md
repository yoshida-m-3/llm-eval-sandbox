# RAG 手法リファレンス

## 0. 基本 RAG パイプライン

すべての手法のベースとなる基本構成。

```mermaid
flowchart LR
    A[ドキュメント] --> B[テキスト分割]
    B --> C[Embedding]
    C --> D[(ベクトルストア)]
    E[質問] --> F[Embedding]
    F --> G{類似度検索}
    D --> G
    G --> H[関連チャンク]
    H --> I[プロンプト\n質問 + コンテキスト]
    E --> I
    I --> J[LLM]
    J --> K[回答]
```

以降の各手法は、このパイプラインの**特定のステージを改善**するもの。

---

## 1. チャンク戦略の改善

テキスト分割 → Embedding の段階を改善する手法群。

### 1-1. チャンクサイズ・オーバーラップ調整

チャンクの大きさと重複幅が検索精度に与える影響を比較する。

```mermaid
flowchart TB
    Doc[ドキュメント]

    subgraph 小チャンク ["小 (chunkSize=50)"]
        S1[チャンク1] --- S2[チャンク2] --- S3[チャンク3] --- S4[...]
    end

    subgraph 中チャンク ["中 (chunkSize=100)"]
        M1[チャンク1] --- M2[チャンク2] --- M3[...]
    end

    subgraph 大チャンク ["大 (chunkSize=200)"]
        L1[チャンク1] --- L2[チャンク2]
    end

    Doc --> 小チャンク
    Doc --> 中チャンク
    Doc --> 大チャンク

    小チャンク --> R1[検索精度: 高\nコンテキスト: 狭い]
    中チャンク --> R2[検索精度: 中\nコンテキスト: 中]
    大チャンク --> R3[検索精度: 低\nコンテキスト: 広い]
```

- **小チャンク**: ピンポイントで該当箇所を見つけやすいが、周辺文脈が欠落する
- **大チャンク**: 文脈は豊富だがノイズも混入しやすい
- **オーバーラップ**: チャンク境界で情報が切れるのを防ぐ

### 1-2. セマンティックチャンキング

文字数ではなく**意味の境界**でテキストを分割する。

```mermaid
flowchart TB
    subgraph 文字数ベース ["RecursiveCharacterTextSplitter（文字数ベース）"]
        direction LR
        C1["東京は日本の首都で、人口は"] --> C2["約1400万人です。\n大阪は商業の"] --> C3["中心地として知られています。"]
    end

    subgraph 意味ベース ["セマンティックチャンキング（意味ベース）"]
        direction LR
        S1["東京は日本の首都で、\n人口は約1400万人です。"] --> S2["大阪は商業の中心地\nとして知られています。"]
    end

    Doc[ドキュメント] --> 文字数ベース
    Doc --> 意味ベース
```

```mermaid
flowchart LR
    A[文1] --> E1[Embedding]
    B[文2] --> E2[Embedding]
    C[文3] --> E3[Embedding]
    E1 --> Sim1{コサイン類似度\n文1↔文2 = 0.92}
    E2 --> Sim1
    E2 --> Sim2{コサイン類似度\n文2↔文3 = 0.41}
    E3 --> Sim2
    Sim1 -->|閾値以上: 同じチャンク| Keep[結合]
    Sim2 -->|閾値以下: 分割| Split[ここで分割]
```

- 隣接する文の Embedding 間のコサイン類似度を計算
- 類似度が閾値を下回った箇所で分割する
- 意味的にまとまった単位でチャンクが生成される

### 1-3. 親子チャンク（ParentDocumentRetriever）

**小さいチャンクで検索精度を確保**しつつ、**大きいチャンクで文脈を提供**する。

```mermaid
flowchart TB
    Doc[ドキュメント]
    Doc -->|大きく分割| P1["親チャンク A\n（500文字）"]
    Doc -->|大きく分割| P2["親チャンク B\n（500文字）"]

    P1 -->|小さく分割| C1["子1"]
    P1 -->|小さく分割| C2["子2"]
    P1 -->|小さく分割| C3["子3"]
    P2 -->|小さく分割| C4["子4"]
    P2 -->|小さく分割| C5["子5"]

    C1 & C2 & C3 & C4 & C5 --> VS[(ベクトルストア)]

    Q[質問] --> VS
    VS -->|"子2 がヒット"| Hit["子2"]
    Hit -->|"子→親を参照"| Return["親チャンク A を取得"]
    Return --> LLM["LLM に親チャンク A を渡して回答生成"]
```

- 子チャンクはベクトルストアに格納（検索用）
- 親チャンクは InMemoryStore に格納（コンテキスト提供用）
- 検索で子がヒット → 対応する親チャンクを LLM に渡す

## 2. 検索の改善

類似度検索 → 関連チャンク取得の段階を改善する手法群。

### 2-1. Top-K 調整

取得件数 k の値が回答品質に与える影響を比較する。

```mermaid
flowchart LR
    Q[質問] --> VS[(ベクトルストア)]

    VS --> K1["k=1\n最も関連する1件"]
    VS --> K2["k=3\n上位3件"]
    VS --> K10["k=10\n上位10件"]

    K1 --> R1["精度: 高\n網羅性: 低\nノイズ: 少"]
    K2 --> R2["精度: 中\n網羅性: 中\nノイズ: 中"]
    K10 --> R3["精度: 低\n網羅性: 高\nノイズ: 多"]
```

- k が小さすぎると必要な情報を取りこぼす
- k が大きすぎると無関係な情報がコンテキストに混入し、回答が劣化する

### 2-2. マルチクエリ（MultiQueryRetriever）

1つの質問を**複数の言い換え**に変換して検索の網羅性を高める。

```mermaid
flowchart TB
    Q[質問:\nPythonでファイルを扱うには？]
    Q --> LLM1[LLM で言い換え生成]

    LLM1 --> Q1["クエリ1:\nPythonのファイル操作方法"]
    LLM1 --> Q2["クエリ2:\nPythonでファイルを読み書きする方法"]
    LLM1 --> Q3["クエリ3:\nPython open関数の使い方"]

    Q1 --> S1[検索結果 A, B]
    Q2 --> S2[検索結果 B, C]
    Q3 --> S3[検索結果 D, A]

    S1 & S2 & S3 --> Merge[重複排除して統合\nA, B, C, D]
    Merge --> LLM2[LLM で回答生成]
```

- 単一クエリでは表現の揺れにより見逃すチャンクを拾える
- 複数クエリの結果を重複排除して統合する

### 2-3. ハイブリッド検索（ベクトル + BM25）

**ベクトル検索**（意味的な類似度）と **BM25**（キーワードマッチ）を組み合わせる。

```mermaid
flowchart TB
    Q[質問] --> V[ベクトル検索\nsimilaritySearch]
    Q --> B[BM25 キーワード検索\nBM25Retriever]

    V --> VR["結果:\n1位: Doc A (score 0.95)\n2位: Doc C (score 0.82)\n3位: Doc E (score 0.78)"]
    B --> BR["結果:\n1位: Doc B (score 8.2)\n2位: Doc A (score 6.1)\n3位: Doc D (score 4.5)"]

    VR & BR --> RRF["RRF（Reciprocal Rank Fusion）\n各検索での順位を統合スコアに変換"]

    RRF --> Final["統合ランキング:\n1位: Doc A（両方で上位）\n2位: Doc B\n3位: Doc C"]
```

```mermaid
flowchart LR
    subgraph VectorSearch ["ベクトル検索が得意"]
        V1["意味的に近い表現\n車 → 自動車"]
    end
    subgraph KeywordSearch ["BM25 が得意"]
        B1["固有名詞・専門用語\nReact 19 → 完全一致"]
    end
    VectorSearch --- Both["組み合わせて\n互いの弱点を補完"]
    KeywordSearch --- Both
```

- ベクトル検索は意味的な類似度は捉えるが、固有名詞の完全一致に弱い
- BM25 はキーワードの一致に強いが、同義語や言い換えに弱い
- RRF: `score = Σ 1/(k + rank)` で各検索結果の順位を統合

### 2-4. リランキング

ベクトル検索の結果を**より精密なモデルで並べ替える** 2段階方式。

```mermaid
flowchart TB
    Q[質問] --> VS["ベクトル検索 k=10"]
    VS --> Candidates["候補 10 件"]
    Candidates --> Scoring["LLM が各候補をスコアリング"]
    Scoring --> Sort["スコア順にソート"]
    Sort --> Top["上位 k 件を選択"]
    Top --> LLM["LLM で回答生成"]
```

#### Bi-Encoder（ベクトル検索）

```mermaid
flowchart LR
    BQ[質問] --> BE1[Encoder] --> BV1["ベクトル"]
    BD[文書] --> BE2[Encoder] --> BV2["ベクトル"]
    BV1 & BV2 --> BCOS["コサイン類似度 → スコア"]
```

- 質問と文書を**別々に**ベクトル化して比較
- 高速・事前計算可。精度は中程度

#### Cross-Encoder

```mermaid
flowchart LR
    CE_IN["質問 + 文書\n（連結して入力）"] --> CE[専用モデル] --> CE_OUT["スコア（直接出力）"]
```

- 質問と文書を**連結して1つのモデル**に入力し、スコアを直接出力
- 高精度だが JS/TS でのローカル実行が困難

#### LLM リランキング（今回採用）

```mermaid
flowchart LR
    LR_IN["プロンプト:\n質問 + 文書\n関連度を1-10で評価して"] --> LR[Ollama LLM] --> LR_OUT["スコア: 8"]
```

- **Bi-Encoder**: 質問と文書を別々にベクトル化→比較。高速だが精度に限界
- **Cross-Encoder**: 質問と文書を連結して1つのモデルに入力→スコアを直接出力。高精度だが JS/TS 環境でのローカル実行が困難
- **LLM リランキング**: 汎用 LLM にプロンプトでスコアリングさせる。Cross-Encoder の役割を LLM で代替

## 3. 回答生成の改善

関連チャンク → LLM → 回答の段階を改善する手法群。

### 3-1. Map-Reduce

各チャンクを**個別に処理**してから結果を**統合**する。

```mermaid
flowchart TB
    Q[質問] --> Search["検索 → チャンク1, 2, 3 を取得"]

    Search --> Map["Map: 各チャンクを個別に LLM で要約\nチャンク1 → 要約1\nチャンク2 → 要約2\nチャンク3 → 要約3"]

    Map --> Reduce["Reduce: 要約1 + 要約2 + 要約3 を\nLLM で統合して最終回答を生成"]

    Reduce --> Answer[最終回答]
```

- 各チャンクを独立に処理するため**並列実行が可能**
- 長文にまたがる情報の統合に有効
- LLM 呼び出し回数が多い（チャンク数 + 1 回）

### 3-2. Refine

チャンクを**順番に読みながら回答を段階的に改善**していく。

```mermaid
flowchart LR
    Q[質問]

    Q --> Step1["LLM:\n質問 + チャンク1\n→ 初期回答"]
    Step1 --> A1[回答 v1]

    A1 --> Step2["LLM:\n質問 + チャンク2\n+ 回答v1 を改善して\n→ 改善回答"]
    Step2 --> A2[回答 v2]

    A2 --> Step3["LLM:\n質問 + チャンク3\n+ 回答v2 を改善して\n→ 改善回答"]
    Step3 --> A3[回答 v3\n＝最終回答]
```

- チャンクを逐次処理するため**前の回答を踏まえた改善**ができる
- 並列実行はできない（逐次処理）
- 後のチャンクほど回答への影響が大きくなるバイアスがある

### Map-Reduce vs Refine 比較

```mermaid
flowchart TB
    subgraph MR ["Map-Reduce"]
        direction TB
        subgraph MapPhase ["Map（並列実行）"]
            MR1["チャンク1 → 要約1"]
            MR2["チャンク2 → 要約2"]
            MR3["チャンク3 → 要約3"]
        end
        MapPhase --> MR_OUT["Reduce: 要約を統合して回答"]
    end

    subgraph RF ["Refine"]
        direction LR
        RF1["チャンク1 → 回答v1"] --> RF2["チャンク2 → 回答v2"] --> RF3["チャンク3 → 回答v3"]
    end
```

|  | Map-Reduce | Refine |
|--|-----------|--------|
| 処理 | 並列処理可能 | 逐次処理のみ |
| 偏り | チャンク間の偏りなし | 後のチャンクにバイアス |
| 弱点 | 統合時に情報が落ちうる | 低速 |
| 適性 | 独立した情報の統合 | 文脈を引き継ぐ改善 |
